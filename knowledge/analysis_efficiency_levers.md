# 分析:世界模型离线训练/在线采样各优化的效率收益测算(2026-07-02)

> 基于实测:模块级参数与 FLOPs 用 `torch.utils.flop_counter` 在真实 646M/128² 模型上取得;
> 吞吐用 runs/mc_d4_b128 实测(340 帧/s,GPU 100%,16.7GB);硬件为 Colab L4
> (bf16 121 TFLOPS / fp8 242 / 显存带宽 ~300GB/s / NVDEC 实测 780 帧/s@1080p / 12 核)。

## 0. 基础账(实测)

| 模块 | 参数 | FLOPs/帧(前向) |
|---|---|---|
| tokenizer 编码器 | 1.6M | 0.49G |
| tokenizer 解码器 | 605M(其中平铺 fc 604M) | 3.10G(fc 占 1.21G) |
| dynamics(8 层时空注意,无 FFN) | 11.9M | 1.27G(解析) |
| shortcut 头 generate@4 步 | ~2M | 0.44G |
| **训练合计(fwd+bwd≈3×)** | 646M | **~15.3G/帧** |

**核心事实:340 帧/s × 15.3G = 5.2 TFLOPS = L4 bf16 峰值的 4.3% MFU。**
训练是**显存带宽/kernel 延迟受限**,不是算力受限 ⇒ 压精度(fp8)几乎无收益,
架构侧(去冗余参数、去像素通路、kernel 融合)才是大头。

## 1. 离线训练侧(基线 340 帧/s)

| 优化 | 预估收益 | 依据 |
|---|---|---|
| **token cache**(冻结 tokenizer 预编码,只训 dynamics) | **4-8×** | 训练 FLOPs 15.3→4.6G/帧;可训参数 646M→14M(AdamW 状态 16.7GB→~2GB ⇒ batch ×4+);数据管线从视频解码变磁盘读 6KB/帧(200MB/s 即 3.3 万帧/s,永不瓶颈) |
| **解码器换卷积头**(除 604M 平铺 fc) | 1.3-1.8× | AdamW fp32 状态省 ~7GB ⇒ batch 16→40;decode 3.1→~1.9G;另有空间归纳偏置的质量收益 |
| torch.compile | 1.2-1.5× | 4.3% MFU 意味着大量小 kernel 延迟,融合正对症 |
| fp8(Ada TE) | 1.0-1.2× | 算力不是束缚,收益近零,**不做** |
| NVDEC 转换管线(已部署) | 2.5×(实测) | 转码 25.6× vs CPU 10.1× 实时,且释放 12 核给 loader |
| motion_sample=4 / hard_weight | 吞吐 0 代价 | 4× 候选解码被闲置 CPU 吸收(实测);收益走样本效率口径 |

**组合(token cache + compile):约 5-10×,训练吞吐 → 2-3k 帧/s,200 小时语料一天可扫多轮。**

## 2. 在线采样侧(想象 rollout;基线 = 现 eval_rollout 朴素实现)

| 优化 | 预估收益 | 依据 |
|---|---|---|
| **隐空间 rollout 不解码**(A 硬约束的免费副产品) | ~2.8× | 每帧省 decode 3.10G,想象只需 dyn+gen 1.71G |
| **KV cache 增量推理** | 6.2×(ctx8/H8) | 现实现每步全量重算:Σ(8+k)=100 次帧前向 vs 缓存后 16 次;horizon 越长收益越大(H=15 时 ~10×) |
| **大 batch 想象** | 5-10× | 隐状态仅 49KB/帧(bf16),batch 1024 轻松放下;将 <5% 利用率推到 ~25% MFU |
| 组合 | **~50-100×** | 25% MFU 下 **~1.7 万想象帧/s**(1.71G/帧) |
| fp8 W 推理(大 batch 后变算力受限才有效) | 1.3-1.8× | L4 fp8 242 TFLOPS |
| NVFP4(租 5090) | 5-10× | FP4 算力 ~14× L4 bf16,带宽 1792 vs 300GB/s=6×;见 design_wca_agent §6 |

**经济学结论:CraftGround 真实环境采样约 20-60 帧/s;优化后想象采样 ~1.7 万帧/s——
想象比真实交互便宜 2-3 个数量级。这是 W/C/A 中 A 靠想象训练的定量根据,
也解释了为何 C(合理性门控)值得做:它守护的是这 300-900× 的杠杆不被 W 的漏洞变成毒药。**

## 3. 优先级排序(按 收益/工作量)

1. token cache(离线 4-8×,同时是多游戏 200h 语料的唯一可行吃法);
2. rollout KV cache + 批量想象(A 阶段前置,50-100×);
3. 解码器卷积头(顺手修 604M 浪费,质量+吞吐双收);
4. torch.compile(一行,1.2-1.5×);
5. fp8/NVFP4:L4 上不做,A 阶段租卡再说。

## 4. 修正:token cache ≠ 架构锁定(用户质疑后补账,2026-07-02)

质疑成立的部分:cache 假设 tokenizer 冻结,而编码方式(乃至换 DINOv3 类语义编码器)
未定。但重建账打消锁定顾虑:编码器仅 1.6M/0.49G 帧,重建瓶颈在视频解码
(360p NVDEC ~3-4k 帧/s)⇒ **200h 全量重建 ≈ 1-2 小时**。规则:
1. mp4+jsonl 永远是事实源,cache 是带 tokenizer 权重 hash 版本号的可丢弃派生物;
2. 划算条件 = 训练加速(每天生效的 4-8×)≫ 重建 2h × 迭代次数(周级离散);
3. 实施时机:tokenizer 本轮迭代(128²/裁剪流)收敛"周级不动"之后。
DINOv3 判断:不会原样成为最终编码器(无解码器,W 需可解码隐空间做流匹配),
但"冻结语义编码器 + 仅供人看的小解码头"(V-JEPA2 式)与 A 禁像素约束天然兼容,
是候选路线;届时切换代价 = 一次 2h cache 重建。

## 5. 更正:NVFP4/FP8 的收益(2026-07-09,RTX 5090 同卡实测)

> 复现:`tests/probe_quant_gemm_5090.py`(vLLM 的 `cutlass_scaled_fp4_mm` + `scaled_fp4_quant`,
> torch `_scaled_mm`)。硬件 RTX 5090 / sm_120。

### 5.1 §2 那行把两个变量混在了一起

> `NVFP4(租 5090) | 5-10× | FP4 算力 ~14× L4 bf16,带宽 1792 vs 300GB/s=6×`

该估算拿**5090 的 FP4** 比 **L4 的 bf16**,里面同时含**换卡**与**换精度**。
光换卡(L4→5090,bf16→bf16)已有 ~3.4× 算力、6× 带宽。**FP4 相对同卡 bf16 的边际收益**才是决策依据。

### 5.2 同卡、同 shape 实测(shape 取自 net/dreamerv3/config.py:deter=512/units=512/feat=1536)

| shape (M,K,N) | bf16 | fp8 | **fp4(含激活量化)** |
|---|---|---|---|
| (1024, 1536, 512) batch 1k | 0.024 ms | 1.54× | **0.58×(更慢)** |
| (4096, 1536, 512) batch 4k | 0.039 ms | 1.71× | 0.98×(打平) |
| (16384, 1536, 512) batch 16k | 0.144 ms | 2.20× | **3.43×** |
| (65536, 1536, 512) batch 64k | 0.505 ms | 2.13× | 1.91× |
| (16384, 512, 512) batch 16k | 0.058 ms | 2.13× | 1.43× |
| (16384, 8192, 8192) LLM 级对照 | 9.09 ms | 1.95× | **5.11×** |

**权重可离线量化一次,激活必须每步现量化。** 把激活量化算进去,数字才可部署。
若只测"权重和激活都预量化"的 kernel 峰值,fp4 看起来有 5.3–5.6×——**那是作弊基准**。

### 5.3 为什么 Dreamer 吃不到 LLM 那 5×

量纲问题:**激活量化开销 ∝ M·K(访存),GEMM 收益 ∝ M·K·N(算力)。**
量化开销只有在 **N 大**时被摊薄。LLM 的 N=8192 ⇒ 5.1×;Dreamer 的 **N=512** ⇒ batch 1k 时反而更慢。

### 5.4 裁决:现在不上,顺序不能颠倒

1. **本仓不是算力受限**(§0 实测 4.3% MFU)。量化提升的是算力上限,对 **kernel 启动延迟一点用没有**。
   对症的是 torch.compile / CUDA graphs / 去小 kernel。
2. §1、§2 已列的 lever(token cache 4-8×、想象 KV cache 6.2×、大 batch 想象 5-10×)
   **必须先做**——它们才是把系统推进"大 batch + 算力受限"区间的东西。
   **那之后**,FP4 才有兑现的前提。先上量化是本末倒置。
3. 兑现时的现实预期:**不是 5-10×,而是 ~2-3.4×**(在 N=512 的 Dreamer GEMM 上),
   除非把量化与前一算子融合、或 N 变大。
4. **若只能选一个,选 FP8**:2.1-2.2× 稳定,`torch._scaled_mm` 原生,per-tensor 缩放,
   量化开销比 FP4 的 gs=16 块缩放小,且无 4bit 的复合误差风险。
5. **精度闸门(design_wca_agent §6 已定,不变)**:H=15 的递归隐空间 rollout 里 4bit 误差会逐步复合。
   量化后必须跑 EV(Δz)/开环优势/recon 与 bf16 对比,过闸才放量。

> 旁证(knowledge/conclusion_omni_nvfp4_5090.md §2):NVIDIA 官方的 30B Omni NVFP4 权重里,
> **只有 MoE 专家(占参数 93%、N 很大)是 4bit**;Mamba 投影是 FP8,SSM 本体(A_log/D/conv1d)
> 干脆保持 BF16。工业界的选择与上面的量纲分析一致——**只把 4bit 用在又大又厚的 GEMM 上。**
