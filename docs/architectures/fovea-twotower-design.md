# 凹视双塔(Fovea-TwoTower)架构设计 v0

**日期**: 2026-07-04
**上游**: [[nemotron-twotower-architecture]](arXiv:2606.26493)、凹式注意力实验(v2/v3, scratchpad/annotate)、gaming-500-hours
**目标平台**: 单卡 3090(24GB), 10Hz 实时

## 0. 核心命题

TwoTower 里"过去以双通道递交:KV(精确、随长度增长)+ Mamba 终态(压缩、常数大小)"。
对本地实时 Agent,**砍掉会增长的那一半,把常数大小的那一半升格为唯一的历史通道**:

> Action 塔每个决策拍的历史信息 = Context 塔逐层 Mamba 状态的播种(O(1),与对局时长无关);
> cross-attention 只查**当前帧**的 token,不查历史 KV。

由此每次决策的显存与计算量均为常数;一场 10 分钟对局和一场 10 秒对局,单拍成本相同。
这是对 TwoTower 状态播种机制的"原生利用"——在 dreamer4 移植表里此机制因无递归状态而
退化为共享 KV;自研骨干装回 Mamba 层后,该机制恢复原形,且恰好是实时 Agent 最需要的性质。

## 1. 总体结构

```
                  10Hz 帧流(1080p)                执行回执(efference copy)
                       │                                  │
       ┌───────────────┼───────────────┐                  │
       ▼               ▼               ▼                  │
  [变化通道]       [YOLO-n 检测]     [HUD 模板OCR]          │
  鼠标dx/dy warp     ~3ms            读数→文本token         │
  自运动补偿帧差       │               │                   │
       │               │               │                   │
       └──► 注视点仲裁 ◄┘               │                   │
             │(运动峰值/准星/新目标竞争)  │                   │
             ▼                         │                   │
  凹区原生裁剪(动态) + 全局tubelet + 符号token ≈ 265 tok/帧   │
                       │                                   │
                       ▼                                   ▼
      ┌─────────────────────────────────────────────────────┐
      │ Context 塔(预训练后冻结, ~1.5B)                        │
      │ 24层混合: 18×Mamba-2 + 6×GQA(每4层1个)                 │
      │ 因果流式吃 [视觉token|动作token|语言token(仅训练)]       │
      │ GQA层: 时间对齐RoPE; Mamba层: Δt嵌入                    │
      │ 产出: 逐层Mamba状态(常数) + 当前帧KV                     │
      └───────────────┬─────────────────────────────────────┘
        状态播种(逐层,i→i)│ 层对齐cross-attn(仅当前帧~265 tok)
                        ▼
      ┌─────────────────────────────────────────────────────┐
      │ Action 塔(可训, 同源初始化; 换游戏=LoRA/克隆此塔)        │
      │ 流匹配去噪一整块动作chunk: H=8拍 × 1 token/拍            │
      │ 动作向量≈20维: (dx,dy)连续 + 16键位 + 滚轮               │
      │ τ条件=adaLN-single; 采样=2~4步Euler(shortcut蒸馏)       │
      └───────────────┬─────────────────────────────────────┘
                      ▼
        10Hz 执行器(逐拍释放chunk) + 重规划触发器
        (周边运动显著/新目标事件 → 中断当前chunk提前重规划)
```

## 2. 组件设计

### 2.1 凹式前端(零学习/轻学习, ~5ms)

| 通道 | 实现 | 输出 | token |
|---|---|---|---|
| 全局 gist | 384×216 缩放, 2帧 tubelet | patch token | ~53 |
| 动态凹区 | 448×252 原生裁剪@注视点 | patch token | ~144 |
| HUD | 固定版式模板 OCR | 文本 token(HP/时间/武器/弹药) | ~20 |
| 物体 | YOLO-n, 词表=自动标注(准星命中校验) | 符号 token(类,x,y,w,h,conf) | ~30 |
| 变化 | 鼠标dx/dy全局warp后帧差, 取峰值 | 符号 token(位置,强度) | ~10 |
| 本体 | 上一拍已执行动作嵌入 | 动作 token | ~8 |

合计 ~265 token/帧。注视点仲裁: 瞄准态跟准星、GUI 态跟鼠标指针、
运动峰值/新检测框按显著度竞争;凹区中心是**每帧被周边选出的动态坐标**,不是屏幕几何中心。
HUD 通道不过 ViT——版式固定的信息用 OCR 转文本是压缩率天花板(138 patch token → ~20 文本 token)。

### 2.2 Context 塔 = 在线世界模型(冻结)

- **配置**: d_model 2048, 24 层(18 Mamba-2 + 6 GQA, 4:1 交错), GQA 16头/4kv头, ~1.5B。
- **输入流**: 按真实时间交错的 (视觉 | 动作 | 语言) token。动作回执进流 = 传出副本
  (efference copy),让世界模型内部自行做自运动扣除;语言流仅训练期存在
  (27B 老师的具身标注按时间戳插入),这是"语言能力继承"的注入口。
- **目标(预训练)**: 下一帧视觉 token 预测(世界动力学) + 标注文本 NLL(语义蒸馏)。
  预训练完**永久冻结**(TwoTower Table 2: 冻结 > 联合微调 26%)。
- **时间**: GQA 层用时间对齐 RoPE(绝对秒); Mamba 层加 Δt 嵌入以容忍变帧率。

### 2.3 Action 塔 = 策略读者(可训)

- 与 Context 塔同构、**同 checkpoint 初始化**(同源方言,第 i 层天然读懂第 i 层)。
- **历史 = 状态播种**: 各 Mamba-2 层初始状态 := Context 塔对应层当前状态。
- **现场 = 层对齐 cross-attn**: 只查当前帧 ~265 token 的 KV。无历史 KV,无翻译层。
- **输出 = 流匹配动作 chunk**: 块 = H=8 拍(0.8s), 每拍 1 token(20 维动作向量投影)。
  高斯插值 x_τ=(1−τ)ε+τz₁, 速度场 MSE, adaLN-single 注 τ(+1.5M 参数)。
  块内双向注意力保证 chunk 时间一致性(压枪 dy 与开火键联合建模,无人为分解顺序)。
- **采样**: 2~4 步 Euler(shortcut 少步蒸馏)。块长设计期钉死——文本域"超训练块长即崩"
  的短板在定长动作域不存在。
- **按键离散化**: 主案=连续输出过 0.5 阈值 + 迟滞;消融=独立 Bernoulli 头。
- **换游戏适应**: 冻 Context 塔,克隆或 LoRA 化 Action 塔,吃新游戏小时级数据。

### 2.4 执行环(闭环补丁)

chunk 内开环是已知风险。重规划触发器:
(a) 固定节奏 2~5Hz 滚动重规划;(b) **事件触发**——变化通道周边运动显著度超阈值、
或 YOLO 出现新高置信目标 → 立即中断当前 chunk 重新去噪。
周边负责"何时看/何时改主意",中央凹负责"看什么"——凹式采样与执行环在此合流。

## 3. 3090 预算

| 项 | 计算 | 延迟 |
|---|---|---|
| 前端(YOLO+帧差+裁剪+OCR) | GPU/CPU 混合 | ~5ms(待实测,偏紧) |
| Context 塔吃一帧 | 2×1.5B×265 ≈ 0.8 TFLOP | 30~55ms @10Hz 每帧 |
| Action 塔一次决策 | 4步×8 tok 去噪(cross 读**预存**的当前帧 KV,不重算)≈ 0.1 TFLOP | 20~30ms @2~5Hz |
| **合计(最坏帧)** | | **~90ms < 100ms**(理想值,见下) |

⚠ 记账警示:(a) Action 塔若每去噪步重算 265 token 的 KV,FLOP 涨到 ~3.3 TFLOP
——实现必须预存 Context 塔当前帧 KV,一次前向四步复用;(b) 265 token 是极小 batch,
实际是访存 bound 而非算力 bound(1.5B bf16 权重单次前向 HBM 读 ~3ms 底线 + 18 层
Mamba scan 串行),表中延迟是理想值,p99 能否守住 100ms **待实测回填**;
(c) 本表只记推理——P0(27B 批量标注)/P1(1.5B×500h 预训练)的训练算力
不在 3090 预算内,平台另议。

显存: 两塔 bf16 共 ~6GB(冻结塔可 4-bit 再省一半) + 激活 + YOLO ≪ 24GB。
状态播种保证显存与单拍延迟**不随对局时长增长**。

## 4. 训练配方

| 阶段 | 内容 | 数据/信号 |
|---|---|---|
| P0 数据工厂 | 27B(vLLM+AWQ 批量)产具身标注; YOLO 词表自动标注(准星命中校验) | 500h 全量 |
| P1 世界预训练 | Context 塔: 下帧预测 + 标注 NLL, 交错流 | 500h 多游戏 |
| P2 策略克隆 | 冻 Context; Action 塔同源初始化, 流匹配 BC 于人类 chunk | 500h 动作对齐 |
| P3 少步蒸馏 | shortcut 蒸馏 4步→2步采样 | P2 模型自蒸馏 |
| P4 换游戏 | 每游戏 LoRA Action 塔 | 新游戏小时级 |

## 5. 闸门实验(按序,过一关开下一关)

1. **Gate A 状态可读性**: 冻结 Context 塔状态 → 线性探针预测未来 0.5s 控制事件
   (开火/鼠标速度档;标签免费来自操作日志——step1 落地口径,非早稿的"敌人在场/HP")。
   **必须带动作泄漏对照**:流含 efference copy,按键强自相关,探针须同时赢
   "单帧"与"单帧+裸动作历史"两个基线,后者不赢 = 状态只是动作缓冲区,世界塔必要性存疑。
   不过关 = 世界模型没把关键信息压进状态,播种无意义。
2. **Gate B 播种消融**: Action 塔 播种 vs 零初始化状态(只给当前帧)。差值 = 本设计核心收益的直接度量。
   各条件固定同一 RNG 种子完全配对(init/数据顺序/流匹配 ε 全同),否则单跑差值可被
   run-to-run 方差伪造;过关后加测**错配播种**(拿别人历史播给你)排除"状态统计先验"解释。
3. **Gate C 冻结消融**: 复刻 Table 2——冻结 vs 联合微调,在动作域验证。
4. **Gate D 跨游戏 IG**: 冻结塔(旧游戏分布)+ 新游戏 Action 塔 相对"从零小塔"的信息增益
   (承接 nemotron 笔记的分布位移风险标注)。

## 6. 风险

- 分布位移: 冻结塔吃旧游戏、Action 塔吃新游戏,"层对齐可读"是否保持 → Gate D。
- 双目标漂移: Action 塔训练后其 Mamba 动力学是否还与播种状态兼容 → 同源初始化+冻结缓解, Gate B 监测。
- 动作域收益倍数: TwoTower 的 2.42× 是文本吞吐,动作域赢面在延迟与联合建模,倍数需重测。
- 按键迟滞阈值与重规划触发阈值是新超参,需在录像回放上离线标定。
