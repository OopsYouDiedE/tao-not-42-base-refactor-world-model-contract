# 凹视双塔·快塔 YOLO 化与"慢塔指导下追踪/导航"设计

> 本文回答三问:(2) Minecraft 分割能否只调向量适配;(3) 快塔能否用 YOLO 作图像解析头;
> (4) 组合后能否在慢塔指导下训练"目标追踪 + 导航"。含设计、验证方案、合格判据、失败对策。
> 结论有实证基础(见 §2 实验)与可复跑探针(`tests/probe_yoloe_minecraft.py`)。
> SSOT:本文只写宏观设计与"为什么";接口/超参以代码为准。

## 0. 现状与资产(勘察结论)

- 慢塔 = 中心凹 = DINO(`net/backbone.py` HF 加载;权重本地缓存 dinov3-vits16),负责 UI/精细语义。
- 快塔 = 边缘视觉 = YOLO,负责粗物体/分割/运动/传 3D。基础设施已在库:
  - `net/encoders/yolo_backbone_encoder.py`:YOLOv8-s + hook P3/P4/P5(15/18/21 层),
    但**只全局池化成一个向量**(丢空间)——追踪/导航要"哪里",需改成保留空间的解析头。
  - `runs/checkpoints/`:`yolov8s.pt`、以及**开放词表分割** `yoloe-11l-seg.pt` /
    `yoloe-26l-seg.pt` / `yoloe-26l-seg-pf.pt`(prompt-free)。ultralytics 8.4.78。
- 确定性 GT 工厂:`tests/integration/collect_s8.py` 的 C2 课程用 `setblock` 把铁矿放在
  **已知相对坐标**(眼平准星中心),`request_raycast=True` 的 `raycast_result` 回传
  **准星所指方块 + 距离**。→ 目标位置/距离**无需人工标注**即可拿到 GT。
- 慢塔→快塔的"指导"通道已存在:`net/bc` 的 `TextCondPolicy`(文本指令向量条件化)、
  `CondPolicy`(标量条件),即本轮清理下沉到 L2 的两类。

## 2. 任务:Minecraft 分割能否"仅调向量"适配

### 实验(`tests/probe_yoloe_minecraft.py`,CPU,s8_full C2 采矿帧)
三种"向量"口径对比,同一批帧:

| 口径 | 机制(调什么) | 结果 |
|---|---|---|
| 文本 prompt | `get_text_pe(["iron ore",...])` 文本嵌入 | 检出稀、置信 0.03–0.15,近乎无效 |
| 视觉 prompt | SAVPE 视觉原型(手框准星中心) | 0 检出(连参考帧自身) |
| prompt-free | 内置大词表(不调向量对照) | **46–48 框、置信 0.36–0.82** |

关键证据(`pf_overlay.png`):prompt-free 把 Minecraft 帧**空间上分割得很好**——墙板、
地面、底部 hotbar 都被正确框成区域(hotbar 被框成 "motel 0.50"),**但每个标签都是真实
世界幻觉**("server room"/"night sky"/"urban"/"darkness")。

### 结论
**难的感知(空间分割/定位)已零样本迁移到 Minecraft;缺的只是"类别/prompt 向量与
Minecraft 的对齐"。** 用户直觉("光调向量就够")**方向正确,但需修正一处**:
现成(零样本)prompt 不迁移——**向量必须用域内示例校准/学习,而非直接拿来用**。

- 因此不需要微调 YOLO 骨干(分割已 work),需要的是:用 GT 自动生成的 Minecraft 示例
  **优化类别 prompt 向量**(文本 PE 或视觉 PE 原型)。这仍属"调向量",非改权重。
- **自动 GT 来源**:setblock 已知坐标 + raycast 确认准星方块 + 方块基色启发(铁矿=灰底
  米色斑点)给出弱标注 → 聚合成每类的视觉原型向量,或训一个线性/prompt 头。

### 若"仅调向量"仍不达标(失败对策,按代价升序)
1. 视觉原型不够 → 学习式 prompt:冻结 YOLOE,只反传 prompt 嵌入/SAVPE 头(几万参数)。
2. 仍不够 → 冻结多尺度特征(P3/P4/P5)上接一个轻量 conv 分割头,用自动 GT 训(仍不动骨干)。
3. 最后手段 → 骨干 LoRA。**预期用不到**(§2 证据:骨干特征已充分且自信)。

### 验证方案与合格判据(预登记)
- 数据:C2 课程 setblock 放铁矿 + 若干干扰块,raycast/坐标出逐帧 GT 框。
- 指标:铁矿类 mask IoU vs 自动 GT;主判据 **mIoU ≥ 0.5** 且显著高于零样本文本 prompt 基线(≥+0.3)。
- 混淆控制:**必须用 640×360 原始渲染帧复测**(本轮用 126px 凹裁剪上采样,偏暗且退化,
  会低估;prompt-free 仍高置信=输入可用,故负结论主因是域/标签而非分辨率,但正式判据要干净帧)。

## 3. 任务:快塔用 YOLO 作图像解析头

### 设计(解析头 = YOLO 检出 → 目标 token 流)
现 `YoloBackboneEncoder` 池化成单向量,丢了"哪里"。追踪/导航需要空间,故解析头改为:

- 每帧 YOLO(E) 分割/检出 → 取 top-K 目标 → 固定 `[K, D]` **目标 token**,
  `D = [cls/n_cls, cx, cy, w, h, conf, area]`(空间归一化 + 语义 + 置信 + 面积≈深度代理)。
- 时序拼成 `[T, K, D]`,**类比 DINO patch token 流喂慢塔那样喂快塔**的因果时序头
  (复用 `net/bc` 的 BCPolicy 干:feat_proj 换成 token 投影,其余不变)。

### 原型验证(已跑通,CPU)
`tests/probe_yoloe_minecraft.py::parse_head` 在 C2 帧上端到端产出 `[T,K=8,D=7]` token 流,
解析管线成立。内容当前偏弱=prompt 未校准(与 §2 一致),**结构/接口已验证**。

### 与 3D 的关系
box 中心/大小 + raycast 距离 → 目标的近似世界坐标,正是"YOLO 传 3D 补 Step1 真空"。

## 4. 任务:慢塔指导下训练"目标追踪 + 导航"

### 网络结构
```
慢塔(DINO/文本)——目标指令编码——► goal 向量 g
                                        │(条件)
每帧 ──► YOLO 解析头 ──► 目标 token [K,D] ─┼─► 因果时序头(BCPolicy 干)──► 动作(相机 dx/dy + 键)
上一步动作 a_{t-1} ─────────────────────┘
```
- goal 向量 g 走 `TextCondPolicy` 同构条件通道(本轮已下沉到 `net/bc`):指令决定"追/去哪个目标"。
- 两功能共用一套快塔,靠 g 切换:
  - **追踪**:让目标 token 的 (cx,cy) 逼近画面中心(相机对准)。
  - **导航**:让目标 box 变大 / raycast 距离变小(靠近)直至到达。

### 数据(GT 驱动的示范工厂,主路;非 VPT)
`collect_s8` 的闭环采矿器**已经**用 raycast 对准并逼近铁矿——它本身就是"追踪+导航"的
脚本示范器。generalize:随机目标位置/类别/干扰 → 采轨迹 → BC 快塔(YOLO token + g → 动作)。
备选:raycast 距离/居中做 reward,走 `rest_update` 的 ReST/GRPO。

### 验证方案与合格判据(自设,预登记)
确定性 GT 免人工:
- **追踪**:随机放目标,量准星-目标角误差。合格 = 200 步中位角误差 ≤ 8°,且相对"冻结相机"
  基线降 ≥40%;**可控性**:指令追 A vs B,被居中的目标随指令切换(切换成功率 ≥0.7)。
- **导航**:合格 = ≥60% 局在 220 步内到达目标 2 格内,且比"恒前进"/"随机"基线 success 高 ≥+0.15。

### 失败分析与对策(依赖前置)
- **强依赖任务2**:目标 token 不可靠则追踪无从谈起 → 先做 §2 向量校准,确保目标类稳定检出;
  或对该目标用视觉原型 prompt。
- 远处目标 box 太小漏检 → 多尺度已在(P3);或训练期加 raycast 作 oracle 辅助通道再蒸馏掉。
- 脚本示范太窄(只会挖正前方)→ 扩目标位置分布 + 加干扰物 + 随机初始朝向。

## 4.5 统一解析头与 G1 校准战役(2026-07-07,代码+实证)

### 统一两能力的形态(net/fovea_twotower/yolo_unified.py)
YOLOE-26 的两项能力(pf 端到端提案 / promptable 零样本向量点名)整合为一个封装模块。
**先勘察后设计的关键发现**:
- **pf 与 promptable 两 checkpoint 骨干非同权**——pf 是词表融合(conv+BN 折叠)后又
  轻微微调的副本(除首层外 cos≈1.0000、相对误差 1–3%,首层漂移大),"一骨干两头"
  的嫁接不成立 → 封装内两通路各跑其训练工况,在 token 接口层融合。
- **打分数学对拍逐字节一致**(tests/probe_yoloe_unified.py V1,max|Δ|=0):
  `score_i = BN_i(cv3_i特征)·w × exp(logit_scale_i) + bias_i`。区域侧只有 BN 无 L2
  (为可融合而设计);因此自算余弦口径 = 单位化(BN(emb))·bank 跨尺度可比。
- **set_classes 的向量原样进 cv4**(V2 对拍):`get_text_pe` 输出已是文本侧终态,
  reprta 只作用于推理期新传 tpe——校准原型与文本 PE 同空间直接互换。
- **陷阱:predict() 会融合**——predictor 把文本向量融进卷积(is_fused),cv3 尾层变
  类卷积、嵌入通道消失。嵌入通路必须直调底层 nn 前向,模块内有护栏 assert。
- 预处理=恒等 letterbox:640×360 上下各垫 12px→640×384,增益 1,全程一个坐标系。
- 输出契约:`forward(img, bank) → tokens [N, 几何6 + C类分数]`,bank 运行时可插拔。

### 确定性 GT 工厂落地(tests/integration/collect_calib640.py + eval_g1.py)
- 采集:640×360 原始帧 + 逐帧位姿(x,y,z,yaw,pitch) + raycast + GT 方块绝对坐标
  (raycast 锚定中心矿反推,不猜 `~` 语义);观察策略不攻击(方块永存);
  铁矿5+煤矿2+泥土2(后两类=硬负类);15 局 750 帧仅数分钟(软渲染比预期快)。
  `--rand_layout` 每局随机布局(反位置先验对照)。
- GT 投影:针孔(竖直 FOV 70°,眼高 1.62,MC yaw/pitch 约定),**只投前脸四角**
  (方块嵌墙,凸包会把不可见侧脸算进 GT=系统性高估)。三重验证:raycast 定量核对
  **178/178** 帧准星落在所指方块投影内;逐像素目检;投影区颜色统计正确。

### G1 迭代记录(判据:铁矿 mIoU≥0.5 且比零样本文本基线 +0.3;每轮先诊断后改)
| 轮次(固定布局内部留出) | 铁 mIoU | 诊断出的病灶 |
|---|---|---|
| 零样本文本基线 | 0.000 | 复现 §2:文本 prompt 域外失准 |
| 盒中心采样+均值原型(提案并集) | 0.032 | 整墙大提案中心恰落在矿上→冒名;36 帧最优提案仅 1 次命名对 |
| 掩膜池化+稠密逐格+均值原型 | 0.178 | 均值原型太弱:格级 acc 0.65 |
| 掩膜池化+稠密逐格+**学习向量** | 0.468 | 残差=8px 格边界锯齿 |
| +概率图双线性上采样 | 0.486 | 同上 |
| ~~2× 分辨率推理~~ | 0.248 | **死路**:放大使纹理尺度偏离骨干训练分布 |
| ~~稠密语义×pf 提案边界融合~~ | 0.356 | **死路**:提案掩膜与前脸 GT 非像素级贴合 |
| +铁类偏置 −1.5(训练侧标定)+翻转 TTA | train 0.566 | 类不平衡致铁过预测,偏置修正 |

### 向量级终审(2026-07-07,runs/g1_yoloe_calib.json,判据先于结果)
测试纪律:固定布局留出集已被多轮变体选择污染 → **终审用 calib640_rand
(随机布局 10 局,采集后从未评过;训练只用固定布局全 15 局)**,n=250 帧:
- 铁 mIoU **0.445**,零样本文本基线 0.0014,**Δ=+0.443**;
- **相对判据 PASS(+0.443 ≥ +0.3,余量 1.5 倍)**;绝对判据 FAIL(0.445 < 0.5,差 0.055)
  → 合取判据记 **FAIL**,按预登记阶梯升级第 2 级(conv 分割头,骨干仍冻结)。
- **反位置先验成立**:随机布局 0.445 vs 固定布局 0.486,无坍塌——学习向量学到的是
  纹理语义非位置记忆(采集器 --rand_layout 对照,铁 5 块只有锚定块固定)。
- 结论定性:"仅调向量"的价值命题(域内校准≫零样本)已实证;差的 0.055 是
  "每格独立分类"的形状表达力上限,非向量质量问题。conv 头(阶梯 2)裁决见下。

### G1 终审:阶梯 2 PASS(2026-07-07,runs/g1_yoloe_conv.json)
ConvSegHead(冻结 P3 嵌入上 3 层 3×3 conv,~0.7M 参数,骨干仍零改动;像素 CE
类逆频权重,6 epoch 数分钟),同一随机布局干净终审口径(n=250):
- **铁 mIoU 0.530 ≥ 0.5 PASS;Δ=+0.528 ≥ +0.3 PASS → G1 整体 PASS**。
- 副类同升:煤 0.389、泥土 0.439(向量级 0.16/0.20)——邻域上下文修的正是边界锯齿。
- 分工定型:**conv 头=已校准核心类的精确掩膜;向量 bank=开放词表扩展名**
  (慢塔运行时新类点名走文本 PE 入 bank,攒够域内样本可再升级为学习向量/新类头)。
  两者共享同一次冻结骨干前向,统一头三通路(提案/嵌入/分割)一次出全。

方法学结论(已站住):
1. **格级线性可分**:P3 单位嵌入上逻辑回归 acc 0.845/铁召回 0.92(均值原型仅 0.65)
   ——信息在冻结骨干的嵌入里,"仅调向量"路线成立,但**提取器要判别式拟合**,
   每类一向量+偏置,仍在阶梯第 1 级(骨干/头全冻结)。
2. **提案召回上限 ~0.49**(十字 GT 被多个提案分割):分割判据用稠密逐格口径
   (=YOLOE cls 分支逐锚语义本义);提案+命名保留为 token 流工况。
3. 采样纪律:嵌入聚合必须掩膜池化(盒中心单点=冒名源);原型采样腐蚀 5px 剔边界格;
   只用 P3(跨尺度 BN 统计不同,混采污染空间)。

### 点名向量的三级来源(同一 bank 互换共存)
1. **域内 GT 锚定学习向量**(主路):GT 掩膜内 P3 格嵌入 → 逻辑回归,每类 512d+偏置;
2. **文本 PE**(新类冷启动):慢塔运行时点名新类的唯一即时通道,先粗召回攒域内样本再升级;
3. SAVPE 视觉原型(备用,零样本 0 检出暂不投入)。

### 与慢塔回路的接口(设计,待闭环验证)
慢塔文本输出有**两个注入点**:① 感知点名——文本 PE 入 bank,快塔 token 即刻多一列
该类分数(慢塔改写快塔"看见什么");② 策略条件——指令嵌入→goal g(改写"追什么")。
两者都是 O(1) 向量插拔,两次慢塔输出间零阶保持,异步不阻塞每帧快塔。
