# 设计：如何吃掉 gaming-500-hours 的高清图像（多游戏预训练管线）

> 2026-07-02。回答"如何更好地吃掉这 ~300h（可用部分）1080p/30fps 游戏录像"。
> 结论先行：**不在 1080p 上直接训练**；高清的价值按模块分层兑现——
> tokenizer 以原生像素密度吃细节，dynamics 以全时长吃动作，解码代价只付一次且付给 NVDEC。

## 0. 现状盘点：我们浪费了多少信息

| 环节 | 分辨率 | 像素量 | 占源比例 |
|---|---|---|---|
| 数据源 | 1920×1080 | 2.07M | 100% |
| 转换落盘（现契约） | 640×360 | 0.23M | 11% |
| **模型输入（img_size 默认 64）** | **64×64** | **4.1K** | **0.2%** |

今天全部四组变体实验都在 64² 上跑。GPU 显存仅用 4.4/23GB。
**最便宜的第一口就是把 img_size 提到 128**（像素 ×4），architecture 零改动。

## 1. 实测硬件事实（2026-07-02，Colab L4，12 核 CPU）

- **NVDEC 硬解码 1080p：25.9× 实时（≈780 帧/s），CPU 占用为零**；解码引擎独立于 SM，
  不与训练算力竞争。CPU 全核解码 1080p 为 29× 实时，但吃光 12 核（= 整个训练数据管线预算）。
- **全 GPU 转码链**（NVDEC 解码→scale_cuda→NVENC 编码 360p）：**25.6× 实时** vs
  现行 libx264 CPU 路径 10.1×，且前者 CPU/SM 双空闲。300h 语料一次性转码 ≈ **12h 墙钟，
  可与训练并行**。注意 NVENC 压缩效率低于 x264（同段 10.3MB vs 6.7MB），需把 cq 调到 ~30
  或接受 ~1.5× 体积。
- 训练数据管线上限 ~4600 帧/s（360p 解码到 64²）；1080p 直接流式训练解码上限只有
  ~780(NVDEC)+870(CPU) ≈ 1600 帧/s，且编码器 FLOPs ×9——**直接吃 1080p 是双输**。

## 2. 核心判断：分辨率的价值按模块分层

- **dynamics 在 16 个 token 的隐空间里学**，看不见像素；它的"分辨率胃口"由 token 容量
  （16×token_dim）封顶。给它更多像素不如给它**更多小时数、更多游戏的动作-动力学对**。
- **高清的信息增量几乎全部属于 tokenizer/decoder**：纹理、字体、准星、粒子——这些决定
  隐码质量，进而决定 dynamics 学到的世界的"分辨率"。
- 隐码容量粗算：16×384 = 6144 维。64²×3=12K 像素（2× 压缩，太浪费容量）；
  128²×3=49K（8×，甜点）；256²×3=197K（33×，隐码开始成为瓶颈，需增 token 或加 conv 级）。
  ⇒ **当前结构下 128² 是不动架构的最优输入档**。

## 3. 三个杠杆（按性价比排序）

### 杠杆 1：img_size 64→128（立即，零改动）
像素信息 ×4，GPU 显存余量足够；62M@batch32 只用 4.4GB。配合放大模型（token_dim/层数
上探 ~150M）同时兑现"加大模型占满 GPU"的标定结论。

### 杠杆 2：一次解码、两路落盘（NVDEC/NVENC 全 GPU 转码）
对每段 1080p 原片，单次 NVDEC 解码后分两路输出，然后删原片（--purge-raw）：
- **路 A｜360p 全景流**（现有 VPT 契约）：dynamics 主粮，动作对齐 jsonl 不变。
- **路 B｜256² 原生密度随机裁剪流**：从 1080p 直接裁 256×256（不缩放），保留原生纹理
  细节，专喂 tokenizer。相机旋转（鼠标）对全屏光流近似均匀，裁剪不破坏运动统计；
  tokenizer 本就是单帧重建，更无所谓。裁剪流只需外观多样性，**取 30-50h 子集即可**
  （外观分布比动力学饱和得快），几 GB 而已。
- ffmpeg 单命令 split filter 即可双路输出；两路都走 NVENC，转换与训练可并行。

### 杠杆 3：token cache（冻结 tokenizer 后预编码，消灭解码+编码器前向）
tokenizer 收敛后冻结，把语料一次性编码为隐 token 落盘。此后 dynamics 训练步
**既无视频解码也无编码器前向**：CPU 瓶颈彻底消失，GPU 算力 100% 喂给放大的 dynamics。
体积：16×384 = 6KB/帧(fp8)。筛选后 200h@15Hz ≈ 10.8M 帧 ≈ **65GB**，可行。
cache 追加式写入，天然断点续传。

## 4. 价值密度过滤：不是 495h 都值得吃

- 帧级：转换时按输入活跃度 + 帧差能量丢弃菜单/挂机/加载段（convert_gaming500 的
  segments 机制已具雏形；batch 内已有 motion_sample=4 二次精选）。
- 游戏级：优先"视角-动作耦合"与 Minecraft 同构的第一/第三人称 3D 游戏
  （valorant 102h、gta-v 35h、palworld 29h、minecraft 41h、cs2 等），
  2D/菜单型游戏降权。估计可用高价值语料 ~250-300h——即本设计标题的"300 小时"。

## 5. 磁盘预算（总余量 174GB）

| 项 | 体积 | 说明 |
|---|---|---|
| 路 A 360p 全景流（300h） | 60-90GB | NVENC cq 调参后趋近 60GB |
| 路 B 256² 裁剪流（40h） | ~8GB | tokenizer 收敛后可删 |
| token cache（200h 筛选, fp8, 15Hz） | ~65GB | 阶段 2 才产生；届时可删部分路 A |
| ckpt + 日志 | ~5GB | 定期传 HF hub（git 不收 runs/） |
| 峰值合计 | ~140-170GB | 紧但可行；原片随转随删是纪律 |

## 6. 执行时序（每步可被 Colab 中断后续传）

1. 转换器改造：cut_video 换 NVDEC/NVENC 双路输出；先转高价值游戏子集。
2. **阶段 1｜tokenizer 多尺度预训练**：360p 全景（缩放到 128²）+ 256² 原生裁剪混采
   → 尺度鲁棒的 tokenizer（顺带兑现动态分辨率目标的第一步）。
3. **阶段 2｜冻结 tokenizer，预编码 token cache**（NVDEC 解码喂编码器，一次性）。
4. **阶段 3｜dynamics 大模型预训练**：~150M+，B 配方（motion_sample=4），cache 直读，
   多游戏 22 维通用动作契约。
5. **阶段 4｜Minecraft 小面微调**：action_proj + AdaLN 调制（~4% 参数），对比
   "从头训 vs 预训练+微调"的 EV/开环差距 → 服务 mine_stone 主线。

## 7. 补充分析：1080p 的像素训练时是否都起作用？（用户问，2026-07-02）

**不能。像素要"起作用"需过三道闸，每道都在扔信息：**

1. **下采样闸（进模型前）**：INTER_AREA 区域平均下每个源像素只贡献局部均值；小于缩放核
   （128² 输入 ≈ 8×15 源像素）的结构——远处矿石纹理、GUI 小字、准星态——不可逆消失。
2. **隐码瓶颈闸**：16 token×384=6144 维封顶。实证锚点：A 组 recon PSNR 仅 26.95dB——
   64² 输入、压缩比才 2× 时 tokenizer 已有损，说明**当前瓶颈是编码器容量而非输入分辨率**；
   此时加大 img_size，多出的像素大部分死在瓶颈上。
3. **损失频谱偏置闸**：流匹配/L2 梯度偏向低频，高频纹理即使进了计算图也几乎不产生梯度。
   "参与计算"≠"驱动学习"。游戏画面冗余极高（h264 可压数百倍），真实信息量远小于像素数。

**推论（修正 §3 杠杆优先级）**：
- 解闸 2 优先：加大 enc_base/token_dim 的优先级 ≥ 加大 img_size，两者应同步（recon dB 为标尺）。
- 闸 1 由 256² 原生裁剪流解：高频以原始形态进编码器，只付 7% 面积代价。
- 闸 3 暂不解：mine_stone 主线要的动力学恰是低频（视角/方块消失/位移），频谱偏置当前是
  帮手；高保真渲染需求出现时再上感知损失/patch 判别器。
- 一句话：不需要所有像素起作用，**要保证"任务相关高频"（方块纹理/准星/快捷栏）有一条
  不经缩放平均的通道进隐码**——即裁剪流的存在理由。

## 8. 风险与开口

- NVENC 转码画质/体积需先在 2-3 段上标定 cq；裁剪流的裁剪位置策略（纯随机 vs
  运动能量加权）待小实验。
- token cache 把 tokenizer 冻死：若阶段 3 发现隐码容量不足（roll_adv 退化），
  需回炉阶段 1 提 token_dim 再重编码——cache 是加速器不是承诺。
- Colab 换机后 NVDEC 可用性需开机自检（ffmpeg -hwaccels 探测，缺则回落 CPU 路径）。

## 9. HDF5 归档管线(用户 2026-07-02 拍板,tests/encode_gaming500_hdf5.py)

- **目标**:全 500h 语料图像 JPEG(360p/15Hz/q80,~20KB/帧)入 HDF5 分片,事件按源生
  30Hz 全率无损双存(结构化数组 gzip + 契约 JSONL gzip 原文),流式上传 HF 私有数据集
  `<user>/gaming500-360p-hdf5`(需 write token)。
- **流控**(用户指定):解码帧积内存缓冲,超 10GB → 线程池 JPEG 压缩追加当前分片;
  分片超 20GB → 封片,后台线程上传 HF 后删本地,另起新片;编码/加载均线程池
  (imencode/imdecode 释放 GIL,线程即真并行)。
- **体量账**:500h ≈ 2700 万帧 ≈ 550GB ≈ 28 片;50MB/s 上传时每片 ~7 分钟,与编码重叠。
- **断点续跑**:manifest 记录"分片→段名"映射(封片时落盘)+ 会话完成态;崩溃残留的
  未封分片启动时校验损坏即清,段不重复编码;上传失败留本地下次补传。
- **与训练契约的关系**:HDF5 归档是"随机访问帧库"(免视频解码),mp4+jsonl 契约
  仍是训练事实源;两者由同一会话/分段逻辑派生(convert_gaming500 按路径 import 复用)。
