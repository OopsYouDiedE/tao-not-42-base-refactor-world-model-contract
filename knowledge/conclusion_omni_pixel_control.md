# 结论:零样本像素直控 Minecraft —— 看得懂,不会动(2026-07-09 实测)

> 协议(用户指定):模型以 **4 fps** 读画面,每帧输出 **5 个动作**填满帧间隔 ⇒ 20 动作/秒。
> Minecraft 1 tick = 50ms ⇒ **1 动作 = 1 tick**,数字严丝合缝。
> 参照 **Lumine**(arXiv:2511.08892,Qwen2-VL-7B 底座玩原神)的配方逐条移植。
> 环境:CraftGround(真 Minecraft Java 版)+ Xvfb,超平坦/生存/和平/固定光照。
> 复现:`tests/probe_omni_minecraft_lumine.py`;原始数据 `docs/results/omni_mc_control/`。
> 模型侧能力见 [conclusion_omni_nvfp4_5090.md](conclusion_omni_nvfp4_5090.md)。

## 1. 验收口径(客观,不靠看图)

生存模式(创造模式放方块不消耗 ⇒ 无法计数),开局给 64 圆石:

```
placed = 64 - inventory["cobblestone"]        # 全局计数,与位置无关
```

另记 3×3×3 `surrounding_blocks` 与 `raycast_result` 供诊断。
任务:在平地上放 ≥4 块相邻圆石,形成平台或短墙。

**先标定测量链路再测模型。** 写死策略扫描(低头角度 × 移动模式)找出真实上限:

| pitch₀ | place_then_strafe | strafe_then_place | back_then_place |
|---|---|---|---|
| 35° | 11 | 12 | 6 |
| 45° | 12 | **12** | 4 |
| 60° | 1 | 6 | 4 |

⇒ 甜区 35–45°,每决策约 1 块(贴放置冷却上限);**低头过度(60°+)是本任务的头号失败模式**。

## 2. 主结果

40 个决策(= 200 tick = 10 s 游戏时间):

| 臂 | 动作表示 | 历史 | 放置数 | 合规率 | 不同动作行 |
|---|---|---|---|---|---|
| **写死 oracle**(2 行动作) | degrees | — | **39** | 1.00 | 2 |
| VLM,pixel 瞄准 | pixel | 8 帧 | **4** | 1.00 | 4 |
| VLM,pixel 瞄准 + 讲清时间线 | pixel | 4 帧 | 3 | 1.00 | **1** |
| VLM,degrees + 脚本代摆相机 45° | degrees | 8 帧 | 3 | 1.00 | 2 |
| VLM,degrees | degrees | 8 帧 | **0** | 1.00 | 1 |
| VLM,27 维离散索引,无历史(v1) | index | 0 | 3 | 1.00 | — |

**模型最好成绩是一个手写两行策略的 ~10%。** oracle 堆出了实体墙(`final_oracle.png`),
证明任务可完成、可测量;模型的失败不是任务设计问题。

## 3. 病灶定位:感知完好,感知→动作不接

同一个模型、同一帧画面:

| 单独问它 | 回答 | 判定 |
|---|---|---|
| 描述画面 | "草地、左侧木屋和树、准星对着空处" | ✅ |
| 准星指向天空/地面/地平线? | `HORIZON` | ✅ |
| 手里拿什么? | `Cobblestone block` | ✅ |
| 红点在哪个像素?(4 组) | 误差 **2.2–5.4 px** | ✅ |
| **要让准星对准地面,该发多少度?**(已明说正=向下) | **`-1`** | ❌ 符号与量级全错 |

### 3.1 连续标定坏,离散绑定好

给两个纯文本示例后,量级立刻正确(`+45`),但**符号仍在 ±45 之间摇摆**;
开 thinking 也一样(`-38` / `45`)。它没有可靠的自我中心上下约定。

而**离散绑定**(哪个键干什么)它毫无问题:W/A/S/D/RMB 语法从不出错,合规率恒为 1.00。

> 这正落在 `design_llm_semantic_layer.md §2` 那张表的分界线上:
> | 归 LLM/文本 | 归 latent WM + 反应头 |
> |---|---|
> | 离散绑定(键→功能) | **连续标定(灵敏度/相机平滑)** |
>
> **该硬边界得到实证支持。** 相机 pitch 的符号与增益,恰是 LLM 拿不下来的那一半。

### 3.2 换成它被训练过的表示,病灶缓解但不消失

标定发现模型用 **1000×1000 归一化像素坐标**(`tests/probe_omni_pointing.py`)。
把动作头从"转多少度"换成"指哪个像素"(再由 FOV 几何换算成相机增量)后:

- 方向符号立刻正确(地面 → Y>500 → 向下);
- 放置数 0 → 4,首次达成任务下限;
- **但仍然坍塌**:每轮吐同一个瞄准点,当作增量累加 ⇒ pitch 撞到 90° 盯住脚下。

### 3.3 决定性证据:控制环里它不看画面

同一 system prompt,喂两张截然不同的画面:

```
地平线(可以建)  ->  <action>500 700 ...>   <action>500 700 ...>
正对脚下(必须抬头) ->  <action>500 700 ...>   <action>500 700 ...>   # 正确答案是 500 250
```

**瞄准点与观测完全无关。** 满屏草地、连地平线都不存在时,它照样说"再往下看"。
`distinct_actions` 一度降到 **1/40** —— 逐字复述 system prompt 里的 worked example。

⇒ 零样本下,它在控制环里退化为**复述最显眼的模板**;相机决策不条件于观测。
不是看不见,是**看见的东西没进到动作里**。

### 3.4 规格写清楚有用,但只解决"形状"不解决"闭环"

初版把 `c1..c5` 只描述为"5 个 tick 的按键",模型把它当成"五个格子挑一个填",
永远只"移一下"或"放一块",从不组合。补上逐拍注释的 worked example 后,
它立刻产出**结构正确**的时间线(`RMB ; - ; D ; D ; RMB` = 放块→冷却→侧移→侧移→再放)。

**形状对了,内容还是死的**——它把那一行抄了 40 遍。
⇒ 提示工程能修"输出格式",修不了"是否条件于观测"。

## 4. 时序:4fps 实时闭环本身是可行的

| | |
|---|---|
| 640×360 一帧 + 短输出,热态全程 | **~0.20 s**(TTFT 0.154 s) |
| 4 fps 预算 | 0.25 s |
| 加 4 帧历史 | p90 ≈ 0.25 s(临界) |
| 加 8 帧历史 | p90 ≈ 0.34–0.37 s(**超预算**) |

"VLM 推理是秒级"的直觉在这里**不成立**:输出只有几十 token 时解码不是瓶颈;
秒级数字来自 thinking 模式的长生成(实测 2–11 s,且会跑飞)。

**真正的实时约束来自历史深度**,而历史深度贵是因为 prefix caching 被关掉了
(见 `conclusion_omni_nvfp4_5090.md §5`)。补上 `SupportsMambaPrefixCaching` 后
这条约束应当基本消失。

环境侧不是瓶颈:CraftGround + llvmpipe 软件渲染在 640×360 下跑 **38.7 tick/s ≈ 2× 实时**
(1280×720 降到 27.2)。**换 GPU 无头渲染对 4fps 控制环无收益**,且会与 vLLM 争显存;
只有大规模离线采集 / 想象 rollout 才值得。

## 5. 裁决与后续

**零样本的 Omni 不能当 Minecraft 的像素控制器。** 这与 Lumine 的前提一致 ——
Lumine 是在原神上**微调**出来的(动作格式 + 显式推理 + 语言接地),
不是靠 prompt 让通用 VLM 直接开。

对本仓的直接含义:

1. `design_llm_semantic_layer.md §1` 否决"LLM 直接指挥 30Hz 动作"的裁决 **实证成立**,不必重开辩论。
2. 慢系统 VLM worker 的定位(episode 级子目标 / 里程碑判定 / 偏好标签)**未被本实验挑战**——
   它的感知、OCR、指点、语义判断全部合格。
3. 若日后仍想走"VLM 直控"这条线,已知的三道门槛按代价排序:
   - (a) **动作头必须用它被训练过的表示**(像素指点 ✅ / 角度 ❌);
   - (b) **需要 SFT 让动作条件于观测** —— 提示工程封顶在 3.3;
   - (c) 相机的连续标定(符号/增益)大概率仍要交给反应头,而非 LLM。
