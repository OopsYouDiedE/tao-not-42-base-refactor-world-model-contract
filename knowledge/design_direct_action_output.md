# 设计探讨:动作直出 vs token 序列化(2026-07-03 第三场辩论)

> **探讨稿——未拍板,待用户裁决。**
> 命题(用户):Agent 与其把鼠标点击序列化成字符 token(如 `{"action":"click","x":512,"y":384}`,
> 约 20–40 token),不如让网络直接输出动作(专用动作头/坐标头/特殊动作 token),token 数低很多,
> 效果或许快很多。按 [design_llm_semantic_layer.md](design_llm_semantic_layer.md) 惯例,
> 正反两轮对抗后收敛。相关:[design_llm_deep_integration.md](design_llm_deep_integration.md)(L2b
> prefill-only 直读状态)。

## 1. 裁决(先行)

命题**部分成立,且必须按场景拆开**:"文本 token vs 直出"不是二选一,是四档谱系(§2);
在我们的 30Hz 快反系统里命题**已是既成事实**,在慢系统里它**不适用**(慢系统本就不该发动作),
在通用 computer-use 场景里正确答案是**分档渐进**——默认约束解码,有数据才上动作头。

## 2. 四档谱系(命题隐藏的连续体)

| 档 | 形态 | 每动作解码成本 | 先例 |
|---|---|---|---|
| R0 | 自由文本 JSON | 20–40 token | 早期 tool-use |
| R1 | 约束解码 + 紧凑 DSL(`click 512 384`) | 6–10 token | function calling / constrained decoding(模式合法性≈免费) |
| R2 | 注册动作 token(动作离散为专用词元) | 1–7 token | RT-2、OpenVLA(256 bin 借稀有词元) |
| R3 | 专用动作头/动作专家(零解码,一次前向出整块动作) | 0 token | VPT 分层头(本仓曾 vendored 的 `net/vpt_lib/action_head.py`[prune3 已删] + `train/minecraft/vpt_action.py` mu-law 11-bin 相机头)、π0 流匹配动作专家、Helix(latent→200Hz 头)、LAPA 潜动作 |

OpenVLA 自身的演化(R2 → OFT 变体的并行解码+连续头,吞吐 25–50×)说明:**延迟真成瓶颈时,
领域自己会从 R2 迁到 R3**——机器人已迁,computer-use 未迁(原因见 §4 反方)。

## 3. token 成本的真实数学

以 ~94 token/s 解码(Nemotron-3-Nano API 实测档)计,单击一次:
- R0:30 token ≈ **320ms**;R1:8 token ≈ **85ms**;R2:2–7 token ≈ 20–75ms;R3:0(随前向摊销)。
- **30Hz 环(66ms 预算)**:R0/R1 不满足预算,R2 勉强,只有 R3 及格——这正是快反头必须是网络头的
  算术根据(第一场辩论红线的另一面)。
- **慢系统 computer-use 环(典型 1.5–3s/步:截屏+视觉 prefill+环境结算)**:R0→R1 省 ~230ms,
  占整环 ~10%(Amdahl)——**动作 token 不是主瓶颈,视觉 prefill 和推理 token 才是**。
  真正改变环结构的是 R3 的**动作分块**(一次前向出 0.5–1s 动作序列,ACT/π0 形态),
  而非省那 30 个 token。
- 费用维度可忽略($0.09/1M token 档,单击 30 token ≈ 三百万分之一美元)。

## 4. 两轮攻防要点

**正方**:延迟算术(§3);坐标精度(回归/分箱头输出校准的连续量+不确定性,文本整数坐标受
BPE 多位数切分影响);动作分块摊销;机器人先例已集体迁 R3。

**反方**(最有杀伤力的第 1 条):
1. **预训练分布对齐**:文本坐标蹭了网络规模的视觉-语言 grounding(截图→文本坐标的数据存量
   巨大);从零训的动作头把这份免费迁移全部丢掉,用数据量和泛化性偿还 token 节省——RT-2 的
   核心发现恰是"token 接口才是网络知识迁移的载体";主流 computer-use(Fuyu/CogAgent 等)
   停在文本坐标,是 grounding 迁移+生态兼容的理性选择,不是没想到。
2. **思维链接地断裂**:推理 token 与动作同流("OK 按钮在右下→点它")保持因果耦合;独立头
   可能与推理脱钩,且丢掉可解析、可校验、可审计的动作日志。
3. **生态**:R3 需要 logits/隐层访问 ⇒ 只能自托管,托管 API 全体出局;全部 agent 工具链
   (重放/安全过滤/框架)说的是文本工具调用。
4. **训练数据**:VPT 为一个 R3 头花了 2000h 承包商真标签+IDM 放大;通用 computer-use 的
   坐标级动作对语料稀缺(markov-ai 这类数据商卖的正是这种燃料——非巧合)。
5. **"token 慢"在异步慢系统里不是瓶颈**(§3 Amdahl):我们已有 L2b(prefill-only 直读状态)
   消灭指导下发的解码;动作直出与 L2b 遵循**同一原则的输入侧与输出侧**——L2b 删输入侧(指导)的语言
   瓶颈,R3 删输出侧(动作)的语言瓶颈,共同原则:**消费者不是人时,别过语言瓶颈**。

## 5. 分场景裁决

- **(a) 本项目 30Hz 快反系统:命题已是既成事实。** actor 就是网络动作头(OneHotDist 直出,
  VPT 分层头是仓内先例),token 从未进 66ms 环(第一场辩论红线)。无事可改。
- **(b) 慢系统 VLM worker:命题不适用,但其精神已被 L2b 采纳。** 慢系统被第一场辩论禁止
  输出低层动作,它直出的应是**指导 latent**(L2b 零解码),动作永远由快反头发。
- **(c) 通用 computer-use agent(用户举例场景):分档渐进。** 默认 R1(约束解码+紧凑 DSL,
  零风险省 ~70% 动作 token);有 1–10 万级动作对数据且可微调时上 R2;仅当延迟 SLO 要求
  >2 动作/s 或需要动作分块、且能自托管+有数据(如采购 markov-ai 语料)时上 R3。
  多数 computer-use 负载的瓶颈在视觉 prefill 与环境等待,升档前先测 Amdahl 占比。

## 6. 红线与执行序(建议,未拍板)

- 不为 R3 头绕开第一场辩论红线:任何形态下慢系统不进 66ms 环、不指挥低层动作;
- R2/R3 若立项,对照臂必须含 R1(约束解码)基线——赢不了 R1 的延迟/成功率综合分即降格;
- 本项目近期无新增工作项:场景 (a) 已完成,(b) 走既有 L2 梯子,(c) 非本仓目标。
